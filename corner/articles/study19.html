解决了什么问题

之前的文章经常提到，代码生成任务的主流做法是根据输入从模型中采样多个输出，然后重排序后选出最可能正确的代码。在重排序阶段，很多方法都是基于测试用例，但是在很多时候测试用例不是那么好获得的，并且大多数方法并不能很好地捕捉执行结果的语义特征。

于是作者提出了 LEVER (Learning to Verify)。具体地说，作者训练了一个验证器 (verifier)，它基于自然语言输入、代码本身和执行结果来确定代码是否正确。通过将验证分数和 LLM 的生成概率相结合，并 marginalize 具有相同执行结果的代码（其实就是合并得分）来对候选代码重排序。

LEVER

设 $x$ 为自然语言输入，生成模型 $P(y\mid x)$ 生成代码 $y$，随后 $y$ 被执行，执行结果为 $\mathcal{E}(y)$，将它们拼接后输入到验证器中，验证器给出一个分数 $P_\theta(v\mid x,y,\mathcal{E}(y))$，它衡量了 $x$ 和 $y$ 匹配的可能性。然后将验证分数和 LLM 的生成概率结合，得到重排概率： $$P_R(y,v_{=1}\mid x)=P_{\rm LM}(y\mid x)\cdot P_\theta(v_{=1}\mid x,y,\mathcal{E}(y))$$ 因为不同的代码可能含有相同的语义，所以作者将执行结果相同的代码的得分合并： $$R(x,y)=\sum_{y'\in S\atop\mathcal{E}(y)=\mathcal{E}(y')}P_R(y',v_{=1}\mid x)$$ 这样就减轻了对代码表面形式的依赖，而更加关注执行结果。

验证器的训练

一般数据集中的元素是三元组 $(x,y,z)$，其中 $z=\mathcal{E}(y)$ 为代码 $y$ 的执行结果，但是很多数据集都只包含执行结果而缺少代码。为了获取训练数据，对于每个输入 $x$，先采样 $n$ 个候选代码 $S=\{y_i\}_{i=1}^n$，然后对于每个 $y_i$，根据比较 $\mathcal{E}(y_i)$ 和 $z$ 来得到其标签 $v=\mathbb{I}(\mathcal{E}(y_i)=z)$，然后将所有的 $(y_i,\mathcal{E}(y_i),v)$ 加入 $x$ 对应的数据集中。对于含有 $y$ 的数据集，将其的标签设为 $1$ 并加入到 $x$ 对应的数据集中，这样就得到了新数据集。

        \begin{algorithm}
            \caption{Create Data}
            \begin{algorithmic}
                \FUNCTION{CreateData}{$D_1,D_2,\cdots,D_N$}
                    \STATE $\hat{D}\gets\varnothing$
                    \FOR{$n=1\to N$}
                        \FOR{$(x,*,z)\in D_n$}
                            \STATE Sample $n$ unique code candidates $S=\{\hat{y}_i\}_{i=1}^n$ from $P_{\rm LM}(y\mid x)$
                            \FOR{$\hat{y}\in S$}
                                \STATE $\hat{z}\gets\mathcal{E}(\hat{y})$
                                \STATE $\hat{D}_x\gets\hat{D}_x\cup\{(\hat{y},\hat{z},\mathbb{I}(\hat{z}=z))\}$
                            \ENDFOR
                            \IF{$D_n$ contains the gold program $y$}
                                \STATE $\hat{D}_x\gets\hat{D}_x\cup\{(y,z,1)\}$
                            \ENDIF
                        \ENDFOR
                    \ENDFOR
                    \STATE\RETURN $\hat{D}$
                \ENDFUNCTION
            \end{algorithmic}
        \end{algorithm}

得到数据集后，对于输入 $x$，训练验证器的目标是最小化负对数似然： $$\mathcal{L}_{\theta}(x)=-\dfrac{1}{|\hat{D}_x|}\sum_{(y,z,v)\in\hat{D}_x}\log P_\theta(v\mid x,y,z)$$

个人看法

整个过程只比朴素做法多了一个 verifier，为什么 verifier 能够利用代码的语义呢？无非就是一个二分类，verifier 模型还要基于不同任务来选择，可扩展性很低。还有一个就是整个方法都基于代码的执行，但是对于有输入输出的代码，测试用例从哪来？这篇文章的实验部分有两个数据集是 SQL 代码，一个是 Math Word Problems 任务，还有一个是 Python 生成（在有测试用例的 MBPP 数据集上实验），如果对于 Python 这类的代码，要是没有测试用例，代码就执行不了，整个方法就瘫痪了，所以感觉很扯，凭什么能中啊？
